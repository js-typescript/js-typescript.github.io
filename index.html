    Type and Interface

    Using type for Data Models

    type is a great choice for complex types like unions or intersections, especially when working with API responses.

    Typescript
    // API Response Type
type Role = "admin" | "editor" | "viewer"; // Union type

type User = {
  id: number;
  name: string;
  email: string;
  role: Role; // Union type used here
  isActive: boolean;
};

// Intersection Type (Combining two models)
type Timestamps = {
  createdAt: string;
  updatedAt: string;
};

type UserWithTimestamps = User & Timestamps; // Combining User and Timestamps

// Example usage
const exampleUser: UserWithTimestamps = {
  id: 1,
  name: "John Doe",
  email: "john.doe@example.com",
  role: "admin",
  isActive: true,
  createdAt: "2025-04-09T12:34:56Z",
  updatedAt: "2025-04-09T15:20:00Z",
};

    Using interface for Component Props

    interface works well for describing props, especially when you want to extend or merge interfaces.

    Typescript
    // Base Props
interface CommonProps {
  id: number;
  isActive: boolean;
}

// Component-Specific Props
interface UserCardProps extends CommonProps {
  name: string;
  role: Role; // Reusing Role type
}

// UserCard Component
const UserCard: React.FC<UserCardProps> = ({ id, name, role, isActive }) => {
  return (
    <div>
      <h3>{name}</h3>
      <p>Role: {role}</p>
      <p>Status: {isActive ? "Active" : "Inactive"}</p>
    </div>
  );
};

// Example usage
const App: React.FC = () => (
  <UserCard id={1} name="John Doe" role="admin" isActive={true} />
);

    Mixing Them in a Single Project

    While it’s common to use both type and interface in the same codebase, here’s a guiding principle:

    • Use type for defining data models, like the shape of API responses.
    • Use interface for defining component props, especially when you anticipate extending or merging definitions.

    Suggested Workflow for Your Project

    Here’s a step-by-step guide tailored to your first project:

    1. Component Props: Use interface for all component props.

      Typescript
      interface ButtonProps {
  label: string;
  onClick: () => void;
  disabled?: boolean;
}

    2. Data Models: Use interface for simple objects and try type for anything complex.

      Typescript
      type ApiResponse = {
  status: "success" | "error";
  data: { id: number; name: string }[];
};

    3. Consistency: Stick to the same convention (e.g., all props use interface) for consistency across the project.

    Example of Gradual Transition

    Here’s how you might evolve your usage of type and interface as your project grows:

    • Start: Use interface for everything:

      Typescript
      interface User {
  id: number;
  name: string;
}

    • Later: Use type when you encounter more advanced scenarios:

      Typescript
      type User = { id: number; name: string } & { email: string };

    As a beginner, it’s perfectly fine to stick to just interface initially to build a strong foundation. As you grow more comfortable, experiment with type in scenarios where its flexibility adds value. The goal is to simplify your learning curve while ensuring your code remains maintainable.
