    Migrating from React JavaScript to TypeScript: A Beginner's Guide

    Transitioning from React with plain JavaScript to TypeScript is a big step, and it's one that brings great long-term benefits (type safety, better IDE support, fewer bugs).

    But yes, there are important things to keep in mind during the switch ‚Äî and I‚Äôll walk you through them as a beginner, in practical, clear terms.

    Overview

    1. You Don‚Äôt Have to Convert Everything at Once

    You can incrementally migrate your codebase. Thanks to this setting in tsconfig.json:

    "allowJs": true

    This allows .js files to coexist with .ts and .tsx files. So start small ‚Äî convert one component, one hook, or one feature at a time.

    2. Understand the File Types

    File Type Purpose
    .ts TypeScript file ‚Äî logic-only (no JSX)
    .tsx TypeScript + JSX ‚Äî used for React components
    .d.ts Declaration files ‚Äî used for defining types for external JS packages (usually not written manually at first)

    ‚ùó Never use .ts for a component. If it has JSX, use .tsx.

    3. Type Your Props (React Components)

    Here‚Äôs how to write component props in TypeScript:

    Old (JS):

    function Button({ label }) {
  return <button>{label}</button>;
}

    New (TS):

    type ButtonProps = {
  label: string;
};

const Button: React.FC<ButtonProps> = ({ label }) => {
  return <button>{label}</button>;
};

    4. Type Your State and Variables

    JS:

    const [amount, setAmount] = useState(0);

    TS:

    const [amount, setAmount] = useState<number>(0);

    But TypeScript can often infer types. For example:

    const [name, setName] = useState("John"); // inferred as string

    Use explicit typing when:

    • You initialize with null or undefined
    • The type is complex (objects, arrays, etc.)
    const [user, setUser] = useState<User | null>(null);

    5. Typing API Responses

    Don‚Äôt just use any. Define types for your API data:

    // types/user.types.ts
export interface User {
  id: number;
  name: string;
  email: string;
}

    Then use it:

    const fetchUser = async (): Promise<User> => {
  const res = await fetch("/api/user");
  return res.json();
};

    6. Avoid any Unless You're Desperate

    TypeScript allows this:

    let data: any;

    But try to avoid it. You lose all type safety. Prefer unknown or actual types.

    7. TypeScript with Events, Forms, Refs, etc.

    These are tricky for beginners ‚Äî here are common ones:

    Event (e.g., onChange):

    const handleChange = (e: React.ChangeEvent<HTMLInputElement>) => {
  console.log(e.target.value);
};

    Form Submit:

    const handleSubmit = (e: React.FormEvent<HTMLFormElement>) => {
  e.preventDefault();
};

    useRef:

    const inputRef = useRef<HTMLInputElement>(null);

    8. TypeScript with Ant Design

    AntD components are already typed. You can often let the type inference work for you:

    <Form.Item name="email" label="Email">
  <Input />
</Form.Item>

    But for custom handlers or complex forms, you might want to define types for form values.

    interface FormValues {
  email: string;
  password: string;
}

    9. Use Type Aliases or Interfaces

    They‚Äôre almost the same for most use cases:

    type User = {
  name: string;
  age: number;
};

interface Product {
  name: string;
  price: number;
}

    Stick to type for data and interface for classes or props, but it‚Äôs mostly preference unless doing advanced stuff.

    ‚úÖ 10. Leverage Your IDE (VS Code)

    TypeScript really shines when you:

    • Hover to see inferred types
    • Get red squiggles for errors
    • Use auto-imports and definitions

    üß† Pro Tip: Set Up Strict Type Checking

    In your tsconfig.json, set:

    "strict": true

    This gives you maximum safety ‚Äî it‚Äôll force you to be explicit and avoid runtime issues.
